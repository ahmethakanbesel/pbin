# Project Research Summary

**Project:** pbin
**Domain:** Self-hosted single-binary file sharing + multi-file transfer + pastebin (Go + SQLite)
**Researched:** 2026-03-19
**Confidence:** HIGH

## Executive Summary

pbin is a unified, self-hosted three-in-one tool that no single competitor currently provides: file sharing, multi-file transfer buckets, and pastebin in one CGO-free Go binary backed by SQLite. The entire Go ecosystem for this class of product is well-understood, with idiomatic patterns (DDD-lite with repository interfaces, embed.FS for the UI, stdlib routing) proven in production at this scale. The recommended approach is a layered architecture — domain packages define repository interfaces, infrastructure packages implement them, and handlers stay thin — so that SQLite can be swapped for Postgres and local disk for S3 without touching business logic. The binary compiles cross-platform without a C toolchain, which is the primary competitive differentiator over every reference product (linx-server, PsiTransfer, Lenpaste, PrivateBin, transfer.sh).

The most important risks all cluster in two areas: SQLite concurrency and security at the file-serving layer. SQLite requires a deliberate two-pool setup (single write connection, multiple read connections) in WAL mode with a busy timeout; skipping this causes intermittent `SQLITE_BUSY` failures under any concurrent load. The file-serving layer must force `Content-Disposition: attachment` on all user uploads and override SVG/HTML content types — serving user-supplied files inline on the same origin enables stored XSS. Both issues are straightforward to prevent if addressed in the foundation phase; both are expensive to retrofit if discovered later.

The feature surface is well-scoped. Table stakes (shareable links, expiry presets, deletion tokens, password protection, one-use flags, REST API, syntax-highlighted paste view, multi-file ZIP download) can all be implemented in a single v1 without external infrastructure. Client-side encryption, per-user accounts, resumable uploads, and thumbnail generation are explicit anti-features for this architecture and should be deferred or consciously excluded. The Unique ID / Slug system and background cleanup goroutine are the two internal foundations everything else depends on.

## Key Findings

### Recommended Stack

The stack is fully stdlib-first Go 1.24 with a minimal set of well-maintained dependencies. `modernc.org/sqlite` (v1.47.0) provides CGO-free SQLite with WAL support — it is the correct choice over `mattn/go-sqlite3` (requires CGO, breaks cross-compile) and `ncruces/go-sqlite3` (wazero-based, more complex dependency graph). Schema migrations use `pressly/goose/v3` embedded via `embed.FS` and applied at startup. Type-safe SQL is generated by `sqlc` at dev time; it is not a runtime dependency. Configuration comes from `knadh/koanf/v2`, which is 3x smaller than Viper and supports TOML file + environment variable overrides. Routing uses Go 1.22+ `net/http.ServeMux` (method + wildcard patterns) — no external framework.

**Core technologies:**
- **Go 1.24**: Language runtime — single binary, zero runtime, cross-compile to any target; generic type aliases and Swiss-table maps stable
- **`modernc.org/sqlite` v1.47.0**: CGO-free SQLite driver — WAL mode, full SQL, cross-compile from any host without a C toolchain
- **`pressly/goose/v3` v3.27.0**: Schema migrations embedded in binary — versioned SQL files, runs at startup automatically
- **`sqlc` v1.30.0** (dev-only codegen): Type-safe SQL queries — eliminates ORM overhead; generated code has zero runtime deps
- **`knadh/koanf/v2` v2.3.3**: Configuration — TOML file + env var overrides, 3x smaller binary than Viper
- **`net/http` ServeMux** (Go 1.22+): Routing — method + wildcard matching covers all ~15 endpoints without an external router
- **`embed` stdlib**: Bundle web UI — `//go:embed` bakes all templates and static assets into the binary at compile time
- **`log/slog` stdlib**: Structured logging — JSON output, levels, zero additional dependencies

### Expected Features

**Must have (table stakes):**
- Shareable link on upload — core purpose; must be immediate, no extra steps
- File expiry with fixed presets (10min, 1h, 6h, 1d, 7d, 30d, 90d, 1y, never) — every reference product has this
- Deletion token returned at upload — users need a way to remove what they uploaded
- One-use / burn-after-reading for files and pastes — PsiTransfer, Lenpaste, PrivateBin all offer this
- Password protection on file shares and transfer buckets — required by privacy-conscious users
- Multi-file transfer bucket with ZIP bundle download — PsiTransfer defines this expectation
- Paste creation with syntax highlighting and raw endpoint — Hastebin/Lenpaste baseline; `/raw/` is critical for CLI workflows
- REST API for all three domains — CLI and integration users depend on this
- Embedded web UI (upload, share, paste view) — required for non-technical users
- Instance-level Basic Auth gating write endpoints — self-hosters need upload access control
- Background expiry cleanup goroutine — without this, expired rows accumulate on disk permanently

**Should have (competitive differentiators):**
- Unified three-in-one service in one binary — no existing tool combines all three domains
- curl-friendly `PUT /<filename>` upload endpoint — transfer.sh/0x0.st define this pattern
- Server-side syntax highlighting (Chroma) for no-JS paste view — Lenpaste proves this is valued
- One-time download on individual file shares (not just pastes) — linx-server and 0x0.st do not offer this

**Defer (v2+):**
- Admin dashboard (view all uploads, force-delete, usage stats) — defer until operators request it
- S3 / object storage backend — defer until local disk proves insufficient at scale
- Webhook on download — polling the metadata endpoint covers the need

### Architecture Approach

The recommended structure follows DDD-lite in Go: three domain packages (`internal/domain/file`, `internal/domain/bucket`, `internal/domain/paste`) each define their entity, repository interface, and service. The infrastructure layer (`internal/storage`) implements all repository interfaces using a shared `*sql.DB` pool; `internal/filestore` implements `StorageBackend` using the local filesystem. Handlers in `internal/handler` are deliberately thin — they decode HTTP requests, call service methods, encode responses. A single background goroutine in `internal/cleanup` sweeps all three domains for expired records. All dependencies are wired in `cmd/pbin/main.go` via constructor injection; no globals. The web UI is bundled via `//go:embed` and served through `http.FileServer(http.FS(...))`. This structure allows SQLite to be swapped for Postgres and local disk for S3 by replacing only the infrastructure layer.

**Major components:**
1. `internal/domain/{file,bucket,paste}` — entity definitions, repository interfaces, business rules (slug generation, expiry validation, one-use logic, password hashing)
2. `internal/storage` + `internal/filestore` — SQLite repository implementations and local filesystem storage, sharing one `*sql.DB` pool
3. `internal/handler` + `internal/cleanup` — HTTP handlers (thin), background expiry worker (crosses all three domains)
4. `cmd/pbin/main.go` — wiring only: config, DB, repositories, services, handlers, routes, cleanup worker, `ListenAndServe`
5. `web/` — embedded HTML templates and static assets; `//go:embed` baked into binary at compile time

### Critical Pitfalls

1. **SQLite write concurrency (`SQLITE_BUSY`)** — Use WAL mode with `PRAGMA busy_timeout=5000`, separate write pool capped at 1 connection (`db.SetMaxOpenConns(1)`), and `BEGIN IMMEDIATE` transactions. Must be set up in the foundation phase before any handler code.

2. **One-time download race condition (TOCTOU)** — Use an atomic `UPDATE ... WHERE downloaded_at IS NULL` and check `RowsAffected == 0` to reject duplicate delivery. Never use SELECT-then-UPDATE for this check.

3. **Stored XSS via uploaded SVG/HTML** — Force `Content-Disposition: attachment` on all served files, override SVG and HTML content types to `application/octet-stream`, set `X-Content-Type-Options: nosniff` and a `Content-Security-Policy` on all pages.

4. **Multipart upload memory exhaustion** — Apply `http.MaxBytesReader` before `ParseMultipartForm`, always `defer r.MultipartForm.RemoveAll()`. For large files, bypass `ParseMultipartForm` and stream directly from `r.Body` to disk.

5. **ZIP bundle download buffering** — Write `archive/zip` directly to `http.ResponseWriter` (which is an `io.Writer`). Never buffer into `bytes.Buffer` — a multi-GB bucket download will OOM the server.

6. **No schema migration strategy** — Use `pressly/goose` embedded migrations from the first commit. Never write `CREATE TABLE IF NOT EXISTS` in Go code. SQLite `ALTER TABLE` limitations make retrofitting migrations extremely painful.

7. **Expiry cleanup goroutine leaking SQL rows** — Always `defer rows.Close()` immediately after `db.QueryContext`. Prefer `db.ExecContext` for bulk deletes (no rows object). Wrap the goroutine in a recover-restart loop. Unclosed rows block WAL checkpointing.

## Implications for Roadmap

Based on all four research files, dependencies flow bottom-up: foundation must be in place before handlers, handlers before UI polish, security hardening woven throughout. The pitfall mapping from PITFALLS.md provides explicit per-phase warnings that directly shape phase boundaries.

### Phase 1: Foundation and Data Layer

**Rationale:** Every feature depends on the database, slug generation, configuration, and file storage. Pitfalls 1, 6, and 7 (SQLite concurrency, migration strategy, goroutine leaks) must all be addressed here — retrofitting any of these is expensive. The architecture's build order (ARCHITECTURE.md) explicitly starts here.

**Delivers:** Working Go module, SQLite with WAL mode + two-pool setup, goose migrations with schema for all three domains, koanf config, crypto/rand slug generation, local filesystem StorageBackend, basic project structure.

**Addresses:** Config, slug generation, database schema (files, buckets, bucket_files, pastes tables), storage backend interface.

**Avoids:** SQLite BUSY errors, broken upgrades from missing migrations, WAL file growing unbounded.

**Research flag:** Standard patterns — well-documented Go + SQLite setup. No deep research needed.

---

### Phase 2: File Upload and Download Handlers

**Rationale:** File sharing is the simplest of the three domains and validates the full upload-store-serve data path before adding bucket and paste complexity. Pitfalls 2, 4, and the path traversal security issue all live here and must be addressed before any other upload surface is built.

**Delivers:** `POST /api/v1/files` multipart upload, `GET /f/{slug}` download, `DELETE /f/{slug}?token=` deletion, expiry preset enforcement, deletion token generation, `Content-Disposition: attachment` on all served files, `http.MaxBytesReader` before parsing, UUID-keyed on-disk storage.

**Addresses:** File share upload, shareable link, expiry, deletion token, direct download URL, one-use file flag.

**Avoids:** Memory exhaustion from multipart, stored XSS via SVG, path traversal via user-supplied filenames.

**Research flag:** Standard patterns — Go multipart upload and file serving are well-documented.

---

### Phase 3: Transfer Bucket and Paste Domains

**Rationale:** Both domains build on the slug + storage foundation from Phases 1-2. The ZIP bundle download (bucket) and one-use TOCTOU fix (paste and file) are the two highest-risk implementation details in the project; tackling them together in a dedicated phase ensures they get proper atomic implementations before any frontend work begins.

**Delivers:** Bucket creation + multi-file upload + streaming ZIP download + password protection; paste creation + raw endpoint + syntax highlighting + one-use atomic delete-on-read; one-time download extended to individual file shares.

**Addresses:** Transfer bucket with ZIP download, paste with syntax highlight + raw endpoint + expiry + one-use, password protection, one-time download on file shares, burn-after-reading semantics.

**Avoids:** ZIP buffer OOM (stream directly to ResponseWriter), TOCTOU on one-use links (atomic UPDATE + RowsAffected), predictable IDs (crypto/rand throughout).

**Research flag:** ZIP streaming is well-documented. One-use atomicity pattern is well-understood but easy to get wrong — follow PITFALLS.md Pitfall 3 exactly.

---

### Phase 4: Expiry Cleanup and Background Worker

**Rationale:** Expiry and the background cleanup goroutine are hard dependencies for the product to be viable in production. The goroutine crosses all three domains and should be built after all three repositories exist. Pitfall 7 (goroutine leaking SQL rows) must be addressed here.

**Delivers:** Background ticker goroutine sweeping all three domains for expired records, disk file deletion coordinated with DB row deletion, recover-restart wrapper, configurable sweep interval, startup disk-free warning, optional total storage quota enforcement.

**Addresses:** Background expiry cleanup goroutine, disk exhaustion prevention (Pitfall 5), WAL checkpoint hygiene.

**Avoids:** `rows` objects left open during cleanup (blocks WAL checkpoint), orphaned on-disk files after DB row deletion, silent goroutine death.

**Research flag:** Standard Go goroutine lifecycle patterns. Follow PITFALLS.md Pitfall 7 checklist.

---

### Phase 5: REST API and Web UI

**Rationale:** Once all three domains are working and expiry is reliable, the REST API surface can be finalized and the embedded web UI built. The API must be consistent across domains before frontend development begins.

**Delivers:** Full REST API (upload, retrieve, delete for files, buckets, pastes), curl-friendly `PUT /<filename>` endpoint, embedded web UI (upload forms, share pages, paste editor with syntax highlighting, bucket view with ZIP download button), `-tags dev` build-tag mechanism for live frontend iteration, `Content-Security-Policy` and security headers on all pages.

**Addresses:** REST API, embedded web UI, curl PUT endpoint, instance-level Basic Auth middleware, security headers.

**Avoids:** Embedded frontend dev workflow pain (Pitfall 9 — add `-tags dev` for disk-serving during development), timing attacks on Basic Auth (subtle.ConstantTimeCompare), CSP gaps.

**Research flag:** Standard Go embed + html/template patterns. highlight.js or Prism for client-side syntax highlighting is well-documented.

---

### Phase Ordering Rationale

- **Bottom-up dependency order** mirrors ARCHITECTURE.md's explicit build order: config → slug → domains → storage → cleanup → handlers → web.
- **Security pitfalls are embedded in the relevant feature phase**, not deferred to a "security sprint" — serving files without `Content-Disposition: attachment` from Phase 2 would require retrofitting every file handler.
- **One-use atomicity and ZIP streaming are in Phase 3**, the phase where they are introduced, not discovered later during load testing.
- **Cleanup worker is Phase 4**, after all three repositories exist, because it depends on all three domain interfaces.
- **Web UI is last** because it is the thinnest layer — all business logic is already tested in Phases 1-4.

### Research Flags

Phases likely needing deeper research during planning:
- **Phase 3:** One-use atomicity with SQLite `DELETE ... RETURNING *` — verify `modernc.org/sqlite` v1.47.0 supports this syntax (SQLite 3.35+ required; bundled version is 3.51.3, so it should work, but confirm in a spike).
- **Phase 5:** Syntax highlighting library choice (highlight.js vs Prism vs server-side Chroma) — tradeoffs depend on no-JS requirement priority; Chroma adds a Go dependency but enables no-JS; highlight.js is client-only.

Phases with standard patterns (skip research-phase):
- **Phase 1:** Go + SQLite WAL setup, goose migrations, koanf config — all have official documentation and confirmed patterns.
- **Phase 2:** Go multipart upload, `http.MaxBytesReader`, `io.Copy` to disk — textbook Go HTTP patterns.
- **Phase 4:** Background goroutine with ticker + stop channel — standard Go concurrency pattern.

### Gaps to Address

- **`modernc.org/sqlite` `DELETE ... RETURNING *` syntax**: Assumed supported because bundled SQLite is 3.51.3 (> 3.35), but should be verified with a minimal spike test before the one-use implementation in Phase 3.
- **Syntax highlighting strategy decision**: Chroma (server-side, Go dep, no-JS support) vs highlight.js (client-only, no Go dep) — tradeoff needs a decision before Phase 5 frontend work. Recommendation: use highlight.js for v1 (zero Go deps), add Chroma in v1.x after confirming no-JS user demand.
- **Rate limiting design**: PITFALLS.md flags per-IP rate limiting as required but no specific library was evaluated. The stdlib does not include a rate limiter; `golang.org/x/time/rate` (token bucket) is the standard Go choice and should be confirmed for use in Phase 5 middleware.
- **Total storage quota implementation**: Two options (running `bytes_used` counter in SQLite vs periodic `du` call) have different tradeoffs under high concurrency. PITFALLS.md recommends the in-SQLite counter; this should be confirmed in Phase 4 design.
